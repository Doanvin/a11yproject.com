Introduction

Where this checklist comes from

The The Web Content Accessibility Guidelines WCAG is a shared standard for web content accessibility that meets the needs of individuals, organizations, and governments. There are three levels of compliance for the WCAG, A, AA, and AAA. These reflect the priority of support:

  1. A: Must support.
  2. AA: Should support.
  3. AAA: May support.

While A support is mandatory, most websites should have a level of AA support to best guarantee support. AAA level compliance is typically reserved for sites that serve a specialized audience.

This checklist is built using WCAG as a reference point. It targets many, but not all level AA issues.

Success criteria

Each item on this checklist corresponds to what is known as a "success criterion," which is a specific rule in the WCAG. Each success criterion is assigned a number and a short title. For example, the rule about text resizing is called 1.4.4 Resize text.

Some accessibility issues may have multiple success criterion apply to them, but we've identified the one we feel is most relevant for each checklist item.

Does this checklist guarantee my site is accessible?

No. However, this does not mean you should not use it.

Working through this checklist will help greatly improve the experience for everyone who uses your site, as the issues it prompts you to check for covers a wide range of disability conditions.

Removing the issues on this checklist helps ensure your site stays flexible and fault-tolerant. There is no such thing as "perfect accessibility" or a site being "100% accessible," and you should be wary of companies and services that promise it.

Need professional accessibility help? Use professional accessibility services.

Required software

These programs can help you uncover accessibility issues in your code, or provide the functionality to determine if assistive technology can actually work with your website.

Automated testing

Automated tests can check your code for common accessibility issues. Be sure to run two or more automated testing tools to ensure comprehensive results, as one automated testing tool may not know to check for some kinds of code that another one does.

  • aXe, by Deque
  • Tenon, by Tenon
  • WAVE, by WebAIM

Screen readers

Screen readers are programs that read the contents of a screen out loud to the person using it. The person can then take action on the items present on the screen, provided the item has been programmed properly.

While not all assistive technology are screen readers, ensuring your site works with one will help most other forms of assistive technology.

Desktop

  • JAWS, with Chrome, Edge, or Internet Explorer on Windows
  • NVDA, with Firefox on Windows
  • VoiceOver, with Safari on macOS

Mobile

  • TalkBack, with Chrome on Android
  • VoiceOver, with Safari on iOS

Content

Content is the most important part of your site.

Use plain language and avoid relying on figures of speech, idioms, and complicated metaphors.
3.1.2 Language of Parts
Write content at an 8th grade reading level.
Ensure that button, a, and label element content is unique and descriptive.
1.3.1 Info and Relationships
Terms like "click here" and "read more" do not provide enough context, especially to someone who is navigating using a list of all buttons or links on a page or view.
Use left-aligned text for left-to-right languages, and right-aligned text for right-to-left languages.
1.4.12 Text Spacing
Centered-aligned or justified text is difficult to read.

Global code

Global code is code that affects your entire website.

Validate the HTML you generate.
4.1.1 Parsing
Valid HTML helps ensure a consistent experience across all browsers and assistive technology.
Use a lang attribute on the html element.
3.1.1 Language of Page
This helps to ensure that screen readers will pronounce all website content correctly.
Provide a unique title for each page or view.
2.4.2 Page Titled
The title element's content, contained in the document's head, is often the first piece of information announced by assistive technology. This helps inform people what screen or view they are going to start navigating.
Ensure that viewport zoom is not disabled.
1.4.4 Resize text
Preventing people from zooming their screen keeps them from being able to increase the size of content on the screen to a point where they can read it.
Use landmark elements to describe the large sections of content on a page.
4.1.2 Name, Role, Value
Use the nav element to wrap site navigation, main for the primary content of the page, etc. This helps to communicate the overall layout of the page, as well as providing quick navigation access.
Ensure a linear content flow.
2.4.3 Focus Order
Remove tabindex attribute values that aren't either 0 or -1. Remove the use of the autofocus attribute.
Remove session timeouts.
2.2.1 Timing Adjustable
If you cannot, let the person using your site know the timeout exists ahead of time, and provide significant notice before the timer runs out.

Keyboard

Ensuring that your site can be used by operated with a keyboard is a very important aspect to making your site accessible.

Ensure there is a visible focus style on interactive elements when they are navigated to via keyboard.
2.4.7 Focus Visible
Can a person navigating with a keyboard or screen reader see where they currently are on the page?
Check to see that keyboard focus order matches the visual layout.
1.3.2 Meaningful Sequence
Can a person navigating with a keyboard or screen reader move around the page in a predictable way?
Remove invisible focusable elements.
2.4.3 Focus Order
Remove the ability to focus on elements that are not visible on the page. This includes things like inactive menus or modals.

Images

Images are a very common part of most websites. There are techniques you can use to help ensure they're enjoyable by all.

All images

Ensure that all img elements have an alt attribute.
1.1.1 Non-text Content.
alt attributes provide a description of an image for people who may not be able to view them. When an alt attribute isn't present on an image, a screen readers may announce the image's file name and path instead.
Make sure that decorative images have empty alt attribute values. Empty alt attributes are also sometimes known as null alt attributes.
1.1.1 Non-text Content
Decorative images do not communicate information that is required to understanding the website's content. Historically they were used for flourishes and spacer gif images, but are less relevant for modern websites.
Provide a text alternative for complex images such as charts, graphs, and maps.
1.1.1 Non-text Content
Is there a plain text listing points on the map or sections of a flowchart? Describe all visible information, including graph axes, data points and labels, and the overall learnings the graphic is communicating.
For images containing text, ensure the alt description includes the image's text.
1.1.1 Non-text Content
For example, the Wikipedia logo should have an alt value of "Wikipedia."

SVG Images

SVG images are images described using markup.

Ensure that svg elements include the code focusable="false" when they are the child element of a focusable element.
1.3.1 Info and Relationships
This prevents a duplicate announcement of focusable items in Internet Explorer.
Add aria-hidden="true" to svg images that are decorative.
4.1.2 Name, Role, Value
This is the equivalent of an empty/null alt value on a non-svg image.
Ensure that svg images that utilize the use element have whitespace between the svg and use elements.
2.1.2 No Keyboard Trap
This solves a bug in Safari which interferes with VoiceOver describing svg images that contain the use element.
Ensure that img elements with a svg source includes the role="img" attribute.
4.1.2 Name, Role, Value
This code is used to convey image semantics correctly for VoiceOver on iOS.

Headings

Headings elements (h1, h2, h3, etc.) help break up the content of the page into related "chunks" of information.

Use headings elements to describe headings.
2.4.6 Headings or Labels
Heading elements should construct a document outline, and not just be used for visual design.
Use only one h1 element per page or view.
2.4.6 Headings or Labels
The h1 element should be used to communicate the high-level purpose of the page or view. Do not use the h1 element for a title that does not change between pages or views (for example, being used for the site's name).
Headings elements should be written in a logical sequence.
2.4.6 Headings or Labels
The order of heading elements should descend, based on the "depth" of the content. For example, a h4 element should not appear on a page before a h3 element.
Don't skip heading levels.
2.4.6 Headings or Labels
For example, don't jump from a h2 to a h4, skipping a h3 element. If heading levels are being skipped to get a specific visual treatment, use CSS classes instead of styling the heading elements directly.

Lists

Lists let people know the content is related, and how many items are included in the grouping.

Use list elements (ol, ul, and dl) for list content.
1.3.1 Info and Relationships
This includes sections of related content, items displayed in a grid, or sibling a elements.

Controls

Controls are interactive elements such as links and buttons that let a person navigate somewhere, or take action on a page or view.

Use the a element for links.
1.3.1 Info and Relationships
Links are used for loading a new page, or shifting keyboard focus from one element to another.
Use the button element for buttons.
1.3.1 Info and Relationships
Buttons are used to submit data or perform an on-screen action which does not shift keyboard focus.
Provide a skip link and ensure that it is visible when focused.
2.4.1 Bypass Blocks
Skip links provide quick access the main page content, allowing a person to quickly and easily bypass repeated content such as a site's main navigation.
Identify links that open in a new window, or links that navigate to external sites.
3.2.2 On Input
Doing so will help people not be surprised by the result of activating the link.

Tables

Tables are a structured set of data that help you understand the relationships between different types of information.

Use the table element to describe tabular data.
1.3.1 Info and Relationships
Does this content look like a table? Use the table element.
Use th for table headers (with appropriate scope attributes).
4.1.1 Parsing
Depending on how complex your table is, you may also consider using scope="col" for column headers, scope="row" for row headers. Many different kinds of assistive technology still use the scope attribute to help them understand and describe the structure of a table.
Use the caption element to provide a title for the table.
2.4.6 Headings or Labels
The table's caption should describe what kind of information the table contains.

Forms

Forms allow people to enter information into a site for processing and manipulation. This includes things like making reservations, placing ordering, and sending messages.

All inputs in a form are associated with a corresponding label element.
3.2.2 On Input
Use a for/id pairing to guarantee the highest degree of browser/assistive technology support.
Use fieldset and legend elements where appropriate.
1.3.1 Info and Relationships
Does your form contain multiple sections of related inputs? Use fieldset to group them, and legend to provide a label for what this section is for.
Inputs use autocomplete where appropriate.
1.3.5 Identify Input Purpose
Providing a mechanism to help people more quickly, easily, and accurately fill in form fields that ask for commonly asked-for information (name, address, phone number, etc.) can be helpful for many kinds of disability conditions.
Ensure that form input errors are displayed in list above the form after the form is submitted.
3.3.1 Error Identification
This provides a way for assistive technology users to quickly have a high-level understanding of what issues are present in the form, especially for larger forms with many inputs. Make sure that each reported error also has a link to the corresponding field with invalid input.
Associate input error messaging with the input it corresponds to.
3.3.1 Error Identification
Techniques such as using aria-describedby allow people who use assistive technology to more easily understand the difference between the input and the error message associated with it.
Ensure that error, warning, and success states are not visually communicated via color alone.
1.4.1 Use of Color
People who are color blind, or who have other low vision conditions may not see the state change, or understand what kind of feedback the state represents if color is the only indicator.
Update the page or view's title to include high-level form feedback.
1.3.1 Info and Relationships
The title element's content is often the first content announced. Adding feedback about the presence of things like errors or warnings allows people to know there are items on the page or view that need their attention.

Media

Media includes content such as pre-recorded and live audio and video.

Make sure that media does not autoplay.
1.4.2 Audio Control
Unexpected video and audio can be distracting and disruptive, especially for certain kinds of cognitive disability such as ADHD. Certain kinds of autoplaying video and animation can be a trigger for vestibular and seizure disorders.
Ensure that media controls are use appropriate markup.
1.3.1 Info and Relationships
Examples include ensuring a video play button has a pressed toggle state when active, or that a volume slider uses <input type="range">.
Check to see that all media can be paused.
2.1.1 Keyboard
Provide a global pause function on any media control. If the device has a keyboard, ensure that pressing the Space key can pause playback.

Video

Confirm the presence of captions.
1.2.2 Captions
Captions allow a person who cannot hear the audio content of a video to still understand its content.
Remove seizure triggers.
2.3.1 Three Flashes or Below Threshold
Certain kinds of strobing or flashing animations will trigger seizures.

Audio

Confirm that transcripts are available.
1.1.1 Non-text Content
Transcripts allow people who cannot hear to still understand the audio content. It also allows people to digest audio content at a pace that is comfortable to them.

Appearance

How your site content looks in any given situation.

Check your content when using browsing modes such as Windows High Contrast Mode or inverted colors.
1.4.1 Use of Color
Is your content still legible? Are your icons, borders, links, form fields, and other content still present? Can you distinguish foreground content from the background?
Increase text size to 200%.
1.4.4 Resize text
Is the content still readable? Does increasing the text size cause content to overlap?
Double-check that good proximity between content is maintained.
1.3.3 Sensory Characteristics
Use the straw test to ensure people who depend on screen zoom software can still easily discover all content.
Make sure color isn't the only way information is conveyed.
1.4.1 Use of Color
Can you still see where links are among body content if everything is grayscale?
Use a simple, straightforward, and consistent layout.
1.4.10 Reflow
A complicated layout can be confusing to understand and use.

Animation

Content that moves, either on its own, or when triggerd by a person activating a control.

Ensure animations are subtle and not too flashy.
2.3.1 Three Flashes or Below Threshold
Certain kinds of strobing or flashing animations will trigger seizures. Others may be distracting and disruptive, especially for certain kinds of cognitive disability such as ADHD.
Provide a mechanism to pause background video.
2.2.2 Pause, Stop, Hide
Background video can be distracting, especially if content is placed over it.
Make sure all animation obeys the prefers-reduced-motion media query.
2.3.3 Animation from Interactions
Remove animations when the "reduce motion" setting is activated. If an animation is necessary to communicating meaning for a concept, slow its duration down.

Color contrast

Color contrast is how legible colors are when placed next to, and on top of each other.

Check the contrast for all normal-sized text.
1.4.3 Contrast
Level AA compliance requires contrast ratio of 4.5:1.
Check the contrast for all large-sized text.
1.4.3 Contrast
Level AA compliance requires contrast ratio of 3.0:1.
Check the contrast for all icons.
1.4.11 Non-text Contrast
Level AA compliance requires contrast ratio of 3.0:1.
Check the contrast of borders for input elements (text input, radio buttons, checkboxes, etc.).
1.4.11 Non-text Contrast
Level AA compliance requires contrast ratio of 3.0:1.
Check text that overlaps images or video.
1.4.3 Contrast
Is text still legible?
Check custom ::selection colors.
1.4.3 Contrast
Is the contrast sufficient? Otherwise someone may not be able read it if they highlight it.

Mobile/Touch

Things to check to best support different device form factors.

Check that the site can be rotated to any orientation.
1.3.4 Orientation
Does the site only allow portrait orientation?
Remove horizontal scrolling.
1.4.10 Reflow
Requiring someone to scroll horizontally can be difficult for some, irritating for all.
Button and link icons can be activated with ease.
2.5.5 Target Size
It's good to make sure things like hamburger menus, social icons, gallery viewers, and other touch controls are usable by a wide range of hand and stylus sizes.
Ensure sufficient space between clickable items in order to provide a scroll area.
2.4.1 Bypass Blocks
Some people who experience hand tremors have a very difficult time scrolling past clickable items which feature zero spacing.
